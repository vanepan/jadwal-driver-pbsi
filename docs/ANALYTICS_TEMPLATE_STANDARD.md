Like every template, this is a pure presentation layer. The view model is
snapshotted from the live analytics compute in
[js/app.js](js/app.js) (`refreshAnalyticsDisplay` → `exportAnalyticsReport`), so
the PDF matches exactly what is on screen — no recomputation, no drift.

**This report does not copy the reimbursement layout.** It *inherits* the header
architecture, footer architecture, branding, typography, and design language, but
defines an analytics-specific, chart-first body — demonstrating the design
system's [presentation-by-document-type](DOCUMENT_DESIGN_SYSTEM.md#presentation-methods)
principle.

---

## Executive Reporting Principles

1. **Headline first.** The first thing management sees is a row of KPI cards.
   Numbers before detail.
2. **Show, don't tabulate.** Comparison and composition are shown as charts
   (bars, pies); tables are secondary and used only for detail.
3. **One accent.** PBSI Red marks the single most important value/series; every
   other element is a muted neutral. The report reads correctly in grayscale.
4. **Deterministic & presentation-friendly.** Charts are vector primitives, so the
   report prints identically everywhere and is clean enough to drop into a
   briefing.
5. **Scope is explicit.** A Filter Snapshot states exactly what data the report
   covers, so figures are never read out of context.

## Layout Overview

A4 portrait, **multi-page is allowed** (unlike the single-page reimbursement
form). Sections are grouped so a heading never orphans and a chart never splits
across a page break.

## Header

The mandatory three-column PBSI header, using the **same positioning and sizing
philosophy as the approved reimbursement template** (logo sized by width to ≈ the

### Right — analytics metadata (right-aligned)
- `Generated At`
- `Generated By`

## Sections

### 1. Executive Summary — *highest visual priority*
Four KPI cards, the first thing management sees:

| Completion Rate | Total Assignments | Driver Bertugas | Kendaraan Digunakan |
|---|---|---|---|
| `{compRate}%` (accent) | `{total}` | `{driverCounts.length}` | `{vehicleCounts.length}` |

### 2. Filter Snapshot
Four inline label/value cells documenting the report scope: **Date Range**,
**Driver**, **Vehicle**, **Bidang**.

### 3. Driver Analytics — **bar chart**
A horizontal bar chart of **assignment count per driver** (top 8), for quick
workload comparison. The most-active driver's bar is accent red; the rest are
muted neutrals. A caption names the **paling aktif** and **paling sedikit** driver.

### 4. Vehicle Analytics — **adaptive: pie or bar**
- **≤ 6 vehicles → pie chart** (composition reads clearly).
- **> 6 vehicles → bar chart** (comparison across many items).

Shows vehicle utilisation, the **most used** vehicle, and **idle** vehicles
(caption).

### 5. Bidang Analytics — **pie chart**
A pie of **request distribution** per bidang (falls back to assignment counts if
no requests), capped at 6 slices with the remainder folded into **Lainnya**. Shows
operational demand composition and the **most active** bidang.

### 6. Completion Quality
Four KPI summary blocks: **Completion Rate**, **Open Rate**, **Completed
Assignments**, **Open Assignments**.

### 7. Data Quality & Alias Resolution — *secondary*
Three KPI blocks (**Alias Terdaftar**, **Warning Teratasi**, **Warning Terbuka** —
accent when > 0) plus a compact **alias resolution** table (type · canonical value
· usage). Rendered only when data-quality data is present.

## KPI Philosophy

- A KPI card is a **bordered box** with a large number (20 pt) over a small dim
  label (7.5 pt).
- **Exactly one** number per card row may use the accent color — reserved for the
  headline metric (Completion Rate).
- KPI cards open Sections 1 and 6 because executives read the number before the
  chart.

---

## Chart Selection Rules

| Section | Data shape | Chart | Why |
|---|---|---|---|
| Driver | many items, magnitude | **Bar** | Compare workload across all drivers. |
| Vehicle | ≤ 6 items | **Pie** | Few categories → composition reads well. |
| Vehicle | > 6 items | **Bar** | Many categories → pie becomes unreadable. |
| Bidang | composition of demand | **Pie** (≤6 + "Lainnya") | Show share of total demand. |

**General rule:** *pie for composition of a small set; bar for comparison across
a large set.* Tables only for detail that a chart cannot convey.

## Filter Snapshot Standard

Every analytics report **must** state its scope. The Filter Snapshot section
echoes the four active analytics filters exactly as applied on screen:

- **Date Range** (e.g., `30 Hari Terakhir`)
- **Driver** (`Semua Driver` when unfiltered)
- **Vehicle** (`Semua Kendaraan` when unfiltered)
- **Bidang** (`Semua Bidang` when unfiltered)

The same period also appears in the header's `Filter Period` metadata line.

## Visual Style

Charts and cards follow the design system's
[Chart Design](DOCUMENT_DESIGN_SYSTEM.md#chart-design) rules:

- Clean, professional, Apple-style, minimalist.
- **Restrained palette:** `#A8292F` (PBSI Red, lead) + muted neutrals
  (`#5A6B7B`, `#7C8A74`, `#C9A66B`, `#8E7A8C`, `#6B8E8A`, `#9A958C`).
- **Avoid:** dashboard clutter, excessive colors, decorative effects, 3D charts,
  gradients, drop shadows.

---

## Tables (Secondary)

Tables are demoted to a supporting role. Use only when a chart cannot convey the
needed detail. Examples: **Top Driver Detail**, **Top Vehicle Detail**, **Alias
Resolution Summary** (the one table currently used, in Section 7).

## Implementation

Built entirely on the existing **Document Generation Foundation** — no new PDF
pipeline:

| Stage | Module |
|---|---|
| Orchestration | [DocumentEngine](js/docs/doc-engine.js) (`generateAndOpen('analytics-report', vm)`) |
| Registration | [TemplateRegistry](js/docs/template-registry.js) (`register('analytics-report', …)`) |
| Rendering | [PdfExporter](js/docs/pdf-exporter.js) (pdfmake backend) |
| Preview | [DocumentViewer](js/docs/document-viewer.js) |
| Print | [PrintManager](js/docs/print-manager.js) |

**Integration points (this redesign):**
- New template [js/docs/templates/analytics-report.js](js/docs/templates/analytics-report.js).
- [js/app.js](js/app.js): import the template (self-register), point
  `exportAnalyticsReport()` at `analytics-report`, and augment the analytics
  snapshot with a compact `dataQuality` summary.
- **Reimbursement template untouched.**

## Known Constraints

- **Charts are vector primitives.** pdfmake's canvas has no native arc, so pie
  slices are drawn as filled polygons (arc sampled into points) and bars as
  rectangles. This keeps output deterministic and grayscale-safe, at the cost of a
  little geometry code in the template.
- **Multi-page is expected.** The report legitimately spans more than one A4 page;
  sections are grouped `unbreakable` so headings and charts never split.
- **Shared logo module.** The template imports the PBSI mark from
  `reimbursement-logo.js`. That module is really the generic PBSI asset; a future
  cleanup should rename it to a neutral `pbsi-logo.js` (or fold it into a shared
  `document-header.js`) so the dependency name isn't reimbursement-specific. See
  [DOCUMENT_DESIGN_SYSTEM.md › Shared Components](DOCUMENT_DESIGN_SYSTEM.md#shared-components).
- **Header/footer are implemented inline** (mirroring reimbursement) because the
  shared `docHeader`/`docFooter` don't yet support a center logo or a two-line
  footer. Extracting `document-header.js` / `document-footer.js` would let both
  baselines inherit them directly.

## Modification Policy

### Allowed (no formal review)
- KPI additions / removals
- Chart tuning (palette order, top-N caps, pie slice cap)
- Adding a secondary detail table
- Adjusting captions / goal lines

### Requires review
- Header redesign
- Footer redesign
- Logo relocation
- Replacing the chart-first approach with tables
- Changing the chart-selection rules (pie/bar thresholds)
