### The pattern (reused, not invented)

New file `js/config/module-loader-registry.js` extends the exact `Map`-cache idiom already proven by `js/workspace/widget-registry.js`'s `GROUP_LOADERS`/`loadGroup()` and `js/exports/export-registry.js`'s `ensureExportModulesLoaded()` — one memoized loader per module, fetched at most once per session.

Two call-site shapes, chosen per how the module is reached:

1. **Explicit navigation** (Dispatch/Executive/Driver Prediction/Driver Wellness/Petty Cash Analytics sections, Vehicle Prediction tab) — the render function became `async`, awaits the loader(s) first, then renders with real data. A brief load moment on first visit is normal for a lazy-loaded section — the same UX the widget groups already established in v1.19.9.

2. **Home's admin briefing** (`buildHomeContext()` → `buildExecutiveDashboardModel()` / `buildExecutiveRecommendations()`) — this is first-paint-critical for every admin session, so it could **not** block on a dynamic import. Instead: `buildDispatchAnalyticsModel()`, `buildPettyAnalyticsModelIfReady()`, and the `exec`/prediction sub-computations stayed fully **synchronous**, using a module-level cache of the resolved engine functions (`_fnComputeDispatchAnalyticsModel` etc.) populated by a background `ensureAdminEnginesLoaded()`. Until it resolves, each degrades to `null` — the *exact* graceful-degradation shape `buildPettyAnalyticsModelIfReady()` already used for "petty store not opened yet" (confirmed by its own pre-existing comment: "a hiccup leaves the field null and the widgets fall back to honest empty states"). Once the background load resolves, it calls the already-existing `refreshHomeWorkspace()` (v1.19.9, no-skeleton-flash in-place refresh) to backfill real data — no new rendering mechanism, no double-render, no first-paint delay.

This means: `buildHomeContext()`, `renderHomeWorkspace()`, and `refreshHomeWorkspace()` needed **zero signature changes** — they're unaware anything became lazy underneath them.

### 3 modules found blocked — left static, not converted

The audit that scoped this phase found 3 of the original 15 candidate modules are still forced in eagerly by a **different**, always-static `app.js` import chain, unrelated to their own listed import line. Converting only their own line would have been cosmetic — zero actual bundle reduction — so they were left as static imports rather than churn code for no measured benefit:

- **`js/analytics/recommendation-accuracy-engine.js`** — forced in via `app.js → decision-replay-drawer.js → decision-replay-service.js`.
- **`js/recommendation/fleet-recommendation-engine.js`** — forced in via two chains: `app.js → recommendation-summary.js`, and `app.js → vehicle-detail-drawer.js → vehicle-recommendation-panel.js`.
- **`js/engineering/analytics/engineering-analytics.js`** — forced in via `app.js → engineering-center.js → engineering-provider.js`, and reachable by both admin *and* every engineering-role session (not admin-exclusive).

Deferring these for real would mean converting `decision-replay-drawer.js`, `recommendation-summary.js`, `vehicle-detail-drawer.js`, and `engineering-center.js`/`engineering-provider.js` — files with much broader blast radius (decision replay and the vehicle detail drawer are reachable from several roles; `engineering-center.js` is the entire Engineering Operations module's mount point). Flagged under Remaining Opportunities below rather than rushed into this pass.

### `analytics-executive-view.js` — confirmed dead code, converted anyway

`analyticsExecMounted` (the flag gating this module's only 2 call sites) is set `true` nowhere in the codebase — a v1.18.8 comment already documented it as retired. Converting its import to lazy means this module now genuinely never loads (previously it loaded on every boot for nothing). No behavior change since the call sites were already unreachable.

### Files touched

`js/config/module-loader-registry.js` (new), `js/app.js` (11 static imports removed, 1 combined dynamic-loader import added, ~10 functions touched).

- `smoke-boot`: PASS, 0 fatal errors (unauthenticated boot — the path every role hits first).

## A2 · Runtime Memory Audit (Objective 4)

Scoped per plan: audited listeners/timers/observers reachable from the A1-converted modules and the general app shell. Shared session-level infra (Firebase assignment/request listeners, push init, H1/H2 reminder timers) stayed explicitly out of scope — cross-role data dependencies where touching them risks the at-most-once/cred-gating invariants documented for the reminder engine.

Findings:

- **The 12 A1-converted modules register zero timers/observers.** Grepped `setInterval`/`ResizeObserver`/`IntersectionObserver`/`MutationObserver` across every `components/`, `analytics/`, `engines/`, `services/` file — none of the 12 match. They're pure render-on-demand functions; A1's conversion introduced no new teardown obligations.
- **`js/analytics/responsive-currency.js`** — already leak-free by explicit design (its own doc comment: "a prior observer is disconnected first... leak-free" — disconnects before reconnecting on every rebind).
- **`js/engineering/diagnostics/engineering-diagnostics.js`** — `refreshTimer` (2s auto-refresh while the diagnostic overlay is open) is correctly `clearInterval`'d on close.
- **`js/app.js` `initV2DriverAvatars()`** MutationObserver — created exactly once at boot (not per-render), session-lifetime by design, not accumulating.
- **`js/timeline.js` `realtimeTimer`** (60s "now" line tick) — starts once per session for any role with Driver-Ops access (admin/driver/bidang) and is never stopped, even if that session spends the rest of its time in Analytics/Engineering and never looks at the timeline again. Bounded (one timer, cheap per-tick cost) — not a leak, but technically "a module not currently visible still consuming a resource." Documented rather than fixed: stopping it would mean touching `setWorkspace()`'s core navigation wiring for a marginal, sub-perceptible gain.
- **`js/components/request-intelligence-panel.js`** (has its own `MutationObserver` on a modal) is genuinely unreachable — not imported by `app.js` at all. Already documented in the codebase's own changelog as known, intentional dead code ("remains in the tree (unmounted)... A future cleanup could remove it"). Costs zero runtime resources today since it never loads; not a new finding, left untouched.

## A3 · Rendering, Navigation & Interaction Audit (Objectives 5–7)

**Real finding, fixed:** the 5 render functions A1 made `async` (`renderVehiclePredictionSection`, `renderDriverWellnessSection`, `renderDispatchAnalyticsSection`, `renderDriverPredictionSection`, `renderExecutiveDashboardSection`) each now have an `await` gap where none existed before. If a user navigates away while the dynamic import is still in flight, the stale render could resolve later and paint into a now-hidden/abandoned container — an offscreen render, exactly what Objective 5 asks to prevent, and a race that didn't exist pre-A1 because these renders used to be fully synchronous.

Fixed by reusing the exact pattern `js/workspace/home-router.js`'s `renderHome()` already established (`host.__wspToken`, "a newer render took over"): each function now stamps a `host.__renderToken` before awaiting, and checks it's still current immediately after — a superseding navigation/re-render wins and the stale one is silently dropped. No new idiom invented.

**Navigation performance (Objective 6):** already correct by construction — `module-loader-registry.js`'s `Map` cache is session-global, so a second visit to any A1-converted section resolves its dynamic import from cache (no re-fetch). Verified structurally (the cache implementation), not re-tested live per the same authenticated-session gap noted in A1.

**Interaction polish (Objective 7):** A1/A3 touched only JS module-loading and one async-race guard — zero CSS/motion surface. v1.20.8's motion tokens and press-feedback rules are untouched and unaffected; no gaps introduced, none audited-and-fixed here.

## A4 · Notification & Offline Runtime Audit (Objectives 8–9)

**Notification Center** (`js/notifications.js`) — already well-designed from v1.20.8 Phase 4, no changes needed:
- `renderNotificationsList()` only runs when the modal is actually open (checked via `style.display === 'flex'` guards at every call site) — not polled.
- `renderActivityLog()` (admin-only) only renders when its modal is explicitly opened, and caps at `allLogs.slice(0, 100)` — already bounded, already lazy.
- The one live subscription (`syncServerNotifications()`, `notifications/{uid}`) runs unconditionally per session — but this is *necessary*, not wasteful: it's what keeps the unread badge count accurate before the panel is ever opened, which is literally what Objective 8 asks for ("Keep unread notifications instantly available").
- **Documented, not fixed:** that subscription has no `limitToLast`/query bound — a very long-lived account could accumulate an ever-growing `notifications/{uid}` node fetched in full on every login. Left as a flagged opportunity rather than touched now — notification/reminder infrastructure has documented subtlety (at-most-once delivery, cred-gating) that a quick "add a query limit" change shouldn't be made without dedicated testing.

**Offline runtime** (`js/app.js` `initOfflineExperience()`, v1.20.8 Phase 8) — already correctly debounced (400ms) and deduped via an `offlineToastShown` flag; registered exactly once at boot. No changes needed.

## A5 · Performance Measurement (Objective 10)

New script: `scripts/runtime-perf-report.mjs` — measures the real, current unauthenticated-boot path (every role hits this before role resolution) via headless Chromium:

| Metric | Measured value |
|---|---|
| JS modules requested at boot | 149 |
| Total JS bytes transferred | 2,756.3 KB |
| First Paint | 732 ms |
| First Contentful Paint | 732 ms |
| DOMContentLoaded | 595 ms |
| A1-converted modules leaked into boot | **0 / 12** |

The 12 modules A1 converted total **315,730 bytes (308.3 KB) / 5,929 lines** measured directly from source. Since these were static top-level imports before A1, they would have appeared in this exact same boot request list — the script explicitly asserts none of their paths are requested, and confirms 0 today. For any non-admin session (driver, bidang, engineering-only roles), this 308.3 KB is not merely deferred — it is never fetched for the lifetime of that session.

**Not measured (honest gap):** an authenticated-admin boot, to show the *background*-loaded engines (dispatch/petty/exec/prediction, ~110 KB more) arriving after first paint rather than blocking it, and the explicit-nav sections' load-on-first-visit behavior. Same root cause as A1's gap — no admin-session puppeteer harness exists yet in this repo.
