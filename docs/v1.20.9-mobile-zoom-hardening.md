# v1.20.9 — Mobile Zoom Hardening

**Sarpras Operations · Engineering Report**

A small, standalone CSS/viewport fix that eliminates iOS Safari's input auto-zoom and disables accidental pinch-zoom, so the PWA stops feeling like a website on a phone. Zero business-logic changes. Ships as its own checkpoint ahead of the larger v1.20.9 Native Runtime Excellence sprint.

**At a glance:** 4 files touched (+ 1 new verification script) · 13/13 new zoom-check assertions pass · 195+ pre-existing regression assertions re-run, 0 failed.

## 1 · Root Cause

**Input auto-zoom.** Safari auto-zooms the viewport when an editable field receives focus with a computed `font-size < 16px`. The codebase already had a fix convention for this — a `@media (max-width: 600px)` block in `style.css` explicitly labeled "iOS Safari zoom prevention" (added pre-v1.20.9) that bumps `.form-group input/select/textarea` to 16px on mobile. The bug was that this convention was **incomplete**, not absent:

- `.date-input` (`style.css:3064`, the assignment-list date filter, a real `input[type="date"]`) sat *inside the very same mobile block* but was excluded — still 11px.
- `.login-card .form-group input` (`platform.css:596`, styling `#loginUsername`) has higher CSS specificity (two classes + element) than the generic mobile-scoped `.form-group input` fix, so it **won the cascade at every viewport width** and silently kept Username at 14px regardless of the "fix" existing elsewhere. `#loginPin` escaped this by luck — it had its own even-more-specific override already set to 16px.
- Several real inputs were never covered by the convention at all: `.comment-input-area textarea` (request/petty-cash reply box, 13px), `.exec-search__input` (executive dashboard search, 14px), `.v2-topbar-search-input` (global topbar search, 12.5px), `.v2-audit-date-input` (admin audit log filter, 12px), `.eng-search-input` (engineering module search, 13px).

**Pinch-zoom.** `index.html`'s viewport meta had no `maximum-scale`/`user-scalable` — nothing restricted it. v1.20.8 deliberately left this alone, calling `user-scalable=no` "accessibility-hostile" and relying on `touch-action: manipulation` (double-tap-zoom + tap-delay only). This sprint reverses that call at the user's explicit direction: iOS's system-level Accessibility Zoom (Settings → Accessibility → Zoom) operates above the browser and is unaffected by the viewport meta, so a zoom path remains for low-vision users even with in-page pinch-zoom disabled.

## 2 · Changes

| File | Change |
|---|---|
| `index.html` | Viewport meta: added `maximum-scale=1.0, user-scalable=no`. Added missing `mobile-web-app-capable` meta (Android/Chrome PWA parity — Apple's own tag was already present). |
| `style.css` | `.date-input` mobile rule: `11px → 16px` (+ `min-width` 78→96px to fit). Added `.comment-input-area textarea { font-size: 16px; }` to the existing mobile zoom-prevention block. Updated the `touch-action: manipulation` comment to note pinch is now handled separately by the viewport meta. |
| `platform.css` | `.login-card .form-group input`: `14px → 16px` (root-cause fix — this specificity-winning rule was the actual reason Username still zoomed). `.v2-audit-date-input`: `12px → 16px` (+ min-width 120→132px). Added mobile-scoped 16px overrides for `.exec-search__input` and `.v2-topbar-search-input`, matching the existing `.pbsi-select-trigger` precedent. |
| `engineering.css` | Added a new `@media (max-width: 600px)` block (none existed in this file before) bumping `.eng-search-input` to 16px. |

No JS changes — pinch/double-tap/gesture prevention is fully handled by the existing `touch-action: manipulation` (unchanged) plus the viewport meta (new); no `gesturestart`/`touchmove` interception was added, since none was needed once the real root causes were fixed.

`manifest.json` and the `apple-mobile-web-app-*` meta tags were audited against Apple's current PWA guidance and left untouched — `display: standalone`, `orientation: portrait-primary`, `theme_color`/`background_color`, and `apple-mobile-web-app-capable` were already correctly configured.

## 3 · Verification

New script: `scripts/zoom-hardening-check.mjs` — headless Chromium at a 390×844 mobile viewport (`isMobile: true, hasTouch: true`), asserting:
- the viewport meta contains `maximum-scale=1.0` and `user-scalable=no`
- `mobile-web-app-capable` is present
- the two live login fields (`#loginUsername`, `#loginPin`) compute to ≥16px font-size
- every other affected field — `.date-input`, `.v2-audit-date-input`, `.comment-input-area textarea`, `.exec-search__input`, `.v2-topbar-search-input`, `.eng-search-input`, and the generic `.form-group input/select/textarea` — computes to ≥16px via synthetic fixtures (since most sit behind auth/role/modal state not reachable from an unauthenticated static-file load)

**Result: 13/13 passed.**

Pre-existing regression suites re-run clean after the change: `smoke-boot` (PASS, 0 fatal errors), `engineering-ui-check` (61/61), `engineering-ui-dom-check` (50/50), `workspace-foundation-check` (24/24), `engineering-runtime-check` (53/53), `request-workflow-check` (11/11), `adaptive-search-check` (10/10).

**Known verification gap.** Puppeteer drives headless Chromium, not WebKit — it cannot literally reproduce Safari's auto-zoom animation to confirm the *behavior* is suppressed, only that the computed styles/meta that *cause* it are corrected. This is the same class of gap v1.20.8 flagged for its swipe gestures: a real iPhone/Safari pass is the only way to close it fully.

---

## 4 · Regression Impact

None expected — every change is either a font-size increase (compensated with matching padding/min-width where the field could visually overflow) or a viewport/meta addition. No selectors were renamed, no DOM structure changed, no JS logic touched. Full suite results above confirm no observed regression.
