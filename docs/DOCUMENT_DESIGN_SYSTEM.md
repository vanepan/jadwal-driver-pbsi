# Document Design System

> Single source of truth for every PDF document generated by the Sarpras
> Operations **Document Generation Foundation**. All current and future
> templates inherit the standards defined here.

| | |
|---|---|
| **Version** | 1.0 |
| **Status** | Active — Foundation Baseline |
| **Platform** | Bidang Sarana dan Prasarana Operations Platform `v{APP_VERSION}` (currently `1.9.3`) |
| **Scope** | All server- and client-generated PDF documents (Reimbursement, Analytics, and future Audit, Asset, Engineering, and AI reports) |
| **First implementation** | [REIMBURSEMENT_TEMPLATE_STANDARD.md](REIMBURSEMENT_TEMPLATE_STANDARD.md) (operational form) |
| **Second implementation** | [ANALYTICS_TEMPLATE_STANDARD.md](ANALYTICS_TEMPLATE_STANDARD.md) (operational report — charts) |

**Purpose** — Define the global document architecture, branding rules, reusable
components, visual language, and generation standards so that every document
looks like it came from the same institution, prints identically on every
device, and can be produced by adding a single template file.

---

## Design Philosophy

Every generated document must be:

- **Professional** — suitable for executives, auditors, and external partners.
- **Institutional** — consistent identity across all report types.
- **PBSI branded** — official PBSI mark and organisational identity in every header.
- **Executive-friendly** — scannable, text-and-tables first, no visual noise.
- **Print-friendly** — A4, deterministic pagination, grayscale-safe.
- **PDF-first** — the PDF is the product; the on-screen preview *is* the export (true WYSIWYG).
- **Deterministic** — identical output across Desktop, Android, iPhone, and PWA. No `html2canvas`, no browser-print re-pagination, no layout drift.
- **Mobile-independent** — layout is defined in points against an A4 page, never derived from the viewport.

## Standard Header (Mandatory)

All documents **must** use the standard three-column header architecture.

```
┌─────────────────────────────┬───────────────┬──────────────────────────────┐
│ Bidang Sarana dan Prasarana │               │           No. Dokumen: ...   │
│ PBSI — Persatuan Bulu        │   [PBSI LOGO] │             Referensi: ...   │
│ Tangkis Seluruh Indonesia    │               │         Tanggal Cetak: ...   │
└─────────────────────────────┴───────────────┴──────────────────────────────┘
                              ──────────────────  (heavy rule)
```

### Left — Organisational identity
- **Line 1:** `Bidang Sarana dan Prasarana` (bold)
- **Line 2:** `PBSI — Persatuan Bulu Tangkis Seluruh Indonesia` (dim)

### Center — PBSI logo
- **Official PBSI asset only** — [assets/Logo-PBSI.png](assets/Logo-PBSI.png).
- **Slightly elevated vertically** — nudged up so it aligns with the first
  organisational line rather than the block centre.
- **Balanced** with the left and right blocks.
- **Must not visually dominate** — sized to ≈ the height of the left text block.

### Right — Metadata block (right-aligned)
A vertical stack of `label: value` lines. Choose the fields relevant to the
document. Examples:
- `No. Dokumen`
- `Referensi`
- `Tanggal Cetak`
- `Generated By`
- `Generated At`

A heavy horizontal rule closes the header band and separates it from the title.

---

## Document Titles

- **Center aligned.**
- **Uppercase.**
- A clear professional hierarchy: a bold uppercase **title**, optionally followed
  by a lighter, sentence-case **subtitle** describing the document's function.

**Examples:**

| Title | Subtitle |
|---|---|
| `FORM REIMBURSEMENT KENDARAAN OPERASIONAL DAN DRIVER` | Formulir Pengajuan Penggantian Biaya Operasional Kendaraan |
| `LAPORAN RINGKASAN ANALITIK OPERASIONAL` | Ringkasan Kinerja Penugasan, Driver, dan Kendaraan |
| `LAPORAN AUDIT SISTEM` | Rekam Jejak Aktivitas dan Perubahan Data |

## Content Layout

**There is no mandatory section structure.** Different reports legitimately need
different layouts, and the design system explicitly allows this flexibility.

| Document | Typical layout |
|---|---|
| **Reimbursement** | Fixed operational form — Sections A–D, signature + receipt area |
| **Analytics** | KPI cards + charts (bar/pie) first, supporting tables secondary; no signature/attachment |
| **Audit** | Chronological tabular ledger |
| **Engineering** | Mixed narrative + specification tables |
| **Asset Management** | Inventory tables + status |
| **AI Reports** | Narrative sections with supporting tables |

Each template owns its internal body. Only the **header**, **footer**, **visual
language**, and **branding** are shared and mandatory.

## Presentation Methods

A document's body may be expressed as **tables**, **visualizations**, or a mix —
whichever communicates the information most clearly. Tables are always valid, but
they are no longer the default for every document.

> **Analytics reports should prefer visual storytelling when it improves
> comprehension.** Forms should prefer structured layouts.

### Approved visualization types

- **KPI Cards** — headline numbers for executive scanning.
- **Bar Charts** — magnitude comparison across many items (e.g., per-driver workload).
- **Pie Charts** — composition / share of a whole (e.g., demand by bidang), for ≤ 6 categories.
- **Trend Charts** — change over time.
- **Summary Visualizations** — compact at-a-glance roll-ups.

### Design goals for visualizations

- **Executive-friendly** — the headline reads in seconds.
- **Management-friendly** — comparisons and composition are obvious.
- **Presentation-friendly** — clean enough to drop into a briefing.

### Presentation by document type

Different document types legitimately use different presentation methods:

| Document type | Primary method | Rationale |
|---|---|---|
| **Operational reports** (Analytics) | **Charts & KPI cards** first; tables secondary | Management needs comprehension at a glance — composition and comparison beat raw rows. |
| **Operational forms** (Reimbursement) | **Structured forms** first; charts rarely if ever | A form captures and validates discrete fields and signatures, not trends. |
| Audit / ledger | Tables | Chronological records are inherently tabular. |

So an **Analytics report may prioritise charts over tables**, while a
**Reimbursement form may prioritise structured fields over charts** — both are
fully compliant with this design system.

## Table Design

Tables are **optional**. When used, they follow a minimalist, "Apple-style" aesthetic:

- Minimalist and professional.
- **Light borders** — thin hairlines (`0.5pt`), soft neutral color.
- **Comfortable but compact** cell spacing.
- **No heavy fills** — reserve subtle fills (`--fill`) for header/total rows only.
- **No excessive grid lines** — let whitespace do the separating.

## Chart Design

When a document uses charts, they must stay within the same restrained visual
language as everything else:

- **Clean, professional, Apple-style, minimalist.**
- **Restrained palette** — PBSI Red for the single most important series/slice,
  muted neutrals for the rest. Grayscale-safe ordering (the largest value reads
  first regardless of color).
- **Avoid:** dashboard clutter, excessive colors, decorative effects, 3D charts,
  drop shadows, gradients.
- Charts are drawn as **deterministic vector primitives** (the PDF pipeline emits
  pure vector shapes — no rasterised chart images), so they print identically on
  every platform.

## Signature Policy

Signature areas are **not globally required**. Include a signature block only when
a document needs a human validation/authorisation mark.

- **Current approved use:** Reimbursement Form.

---

## Attachment Policy

Attachment areas are **not globally required**. Include a dashed attachment area
only when physical evidence must be affixed.

- **Current approved use:** Reimbursement Form.
- **Examples of attachments:** fuel receipts, toll receipts, parking receipts.

---

## Footer (Mandatory)

All documents use the standard footer, rendered in the bottom page margin in a
small, faint, neutral type.

```
Document Name                                                    Hal. X / Y
PBSI Operations Platform v{APP_VERSION}
```

### Left (two lines)
- **Line 1:** the document's name.
- **Line 2:** `PBSI Operations Platform v{APP_VERSION}` (`APP_VERSION` injected from [js/config.js](js/config.js)).

### Right
- `Hal. X / Y` — current page / total pages.

## Color Policy

| Role | Token | Value | Usage |
|---|---|---|---|
| **Primary** | `--accent` | PBSI Red `#A8292F` | Sparingly — status emphasis, key highlights |
| **Ink** | `--ink` | `#1A1917` | Body text, rules |
| **Secondary / dim** | `--dim` | Neutral Gray `#5B5953` | Labels, subtitles |
| **Faint** | `--faint` | `#94918B` | Footer, captions |
| **Line** | `--line` | `#C9C6C0` | Borders |
| **Fill** | `--fill` | `#F7F6F3` | Subtle header/total fills |

**Rules:**
- Professional and restrained.
- **Grayscale-friendly** — every document must read correctly when printed in black and white.
- Avoid decorative colors.
- Avoid large color blocks.

Tokens are defined centrally in [js/docs/doc-theme.js](js/docs/doc-theme.js).

## Shared Components

The design language is delivered through shared, reusable builders. Today these
live as node-builder functions and tokens inside
[js/docs/doc-theme.js](js/docs/doc-theme.js) (`docHeader`, `headerRule`,
`docFooter`, `tableLayout`, `TOKENS`).

As more templates are added, these should be extracted into dedicated,
independently-versioned component modules:

| Future component | Responsibility |
|---|---|
| `document-header.js` | The mandatory three-column header, **including a center-logo capability**. |
| `document-footer.js` | The mandatory two-line-left / page-number-right footer. |
| `metadata-block.js` | The right-aligned `label: value` metadata stack. |
| `section-title.js` | Standardised uppercase section labels. |
| `table-styles.js` | The minimalist table layout (`tableLayout` + a dense variant). |

> **Note (current reality):** the shared `docHeader` does not yet support a
> center logo and `docFooter` is single-line. The Reimbursement template
> therefore implements its header and footer **inline** to meet this standard.
> Extracting a logo-capable `document-header.js` and a two-line
> `document-footer.js` is the recommended next refactor so all templates inherit
> them directly. See [REIMBURSEMENT_TEMPLATE_STANDARD.md](REIMBURSEMENT_TEMPLATE_STANDARD.md#known-constraints).

## Document Generation Foundation

The runtime architecture that turns a template + data into a viewable,
printable, downloadable PDF. Callers touch only `DocumentEngine`.

| Module | File | Responsibility |
|---|---|---|
| **DocumentEngine** | [js/docs/doc-engine.js](js/docs/doc-engine.js) | Orchestrator and single entry point. `generate(templateId, data)` resolves the template, builds the pdfmake definition, renders a real PDF **Blob** via the exporter, and caches the result per `(templateId + data)`. `open()` shows the reusable viewer. Backend-agnostic. |
| **TemplateRegistry** | [js/docs/template-registry.js](js/docs/template-registry.js) | Maps a template `id` → descriptor `{ build, filename, meta }`. Templates **self-register** on import (side-effect). `getTemplate(id)` / `listTemplates()`. |
| **PdfExporter** | [js/docs/pdf-exporter.js](js/docs/pdf-exporter.js) | Pluggable PDF backends. Default `pdfmake` (client-side; lazy-loads pdfmake + fonts). A `puppeteer` (server-side) seam is reserved for the future. `exportToPdf(definition) → Blob`. |
| **DocumentViewer** | [js/docs/document-viewer.js](js/docs/document-viewer.js) | One reusable modal that renders the real PDF blob (**WYSIWYG: the preview is the export**) with Print / Download / Share. Injects its own DOM + styles once; owns the object-URL lifecycle. |
| **PrintManager** | [js/docs/print-manager.js](js/docs/print-manager.js) | Prints the already-paginated PDF blob deterministically (hidden iframe; iOS falls back to a native-viewer tab). Because the source is pre-paginated, the browser can only scale-to-page — never re-paginate — which is what guarantees cross-platform consistency. |

**Adding a new report** = one template file + one `register()` call + one
integration point. No new PDF pipeline.

## Future Guidance

All future templates **must inherit**:

- **Header standards** — the mandatory three-column branded header.
- **Footer standards** — the mandatory two-line-left / page-number-right footer.
- **Design language** — tokens, color policy, minimalist tables, grayscale safety.

…while remaining **free to define their own internal body layout** appropriate to
the report (see [Content Layout](#content-layout)).

When a new document needs a signature or attachment area, follow the patterns
established by the Reimbursement baseline rather than inventing new ones.
