---

## 1. Research Findings (Phase A)

Accepted approaches reviewed across Fleet Management, Transportation/Logistics Operations,
Workforce Scheduling, and Driver-Utilization analytics:

| Approach | Source domain | Takeaway applied here |
|---|---|---|
| **Composite indicator** (normalize → weight → aggregate) | OECD *Handbook on Constructing Composite Indicators* (2008) | Heterogeneous units (count, hours, km) at different magnitudes must be **normalized before** they are combined. Never sum raw values. |
| **Hours-of-Service / driver utilization** | FMCSA HOS; telematics (Samsara, Geotab) | **Engaged/working hours** is the most direct, time-true measure of driver workload — the dominant operational metric. |
| **Distance as exposure** | Logistics & insurance/safety analytics | **Kilometres travelled** is the primary *exposure* measure (wear, fuel, risk). "1 trip ≠ 1 trip". |
| **Task/assignment volume** | Workforce scheduling / dispatch | Task **count** captures fixed per-task coordination overhead, but over-weights short tasks; weakest effort proxy → lowest weight. |

**Why not assignment count alone** (the spec's motivating case): Driver A (20 asg / 40 h / 300 km)
vs Driver B (15 asg / 90 h / 1 500 km). Count ranks A higher; hours+distance rank B higher.
A credible workload measure must weigh all three at once — exactly what a composite index does.

**Why max-normalization (not z-scores):** the reference is the *busiest driver in the same
period*, so a score reads as "share of the period's hardest-working driver". It is bounded
[0,100], never negative, and directly explainable to operations users — no standard-deviation
literacy required. This is the OECD "distance-to-reference (max)" method.

## 2. Formula (Phases B & H)

For each driver in the selected period, over their **completed** assignments:

```
raw inputs:   A = completed assignments
              H = actual working hours      (Σ actualHours, from startedAt→completedAt)
              D = distance travelled (km)    (Σ distanceTravelled, odometer; 0 if no vehicle)

normalize against the cohort max (the period's busiest driver):
  assignmentIndex = A / maxA × 100      (0 when maxA = 0)
  hoursIndex      = H / maxH × 100      (0 when maxH = 0)
  distanceIndex   = D / maxD × 100      (0 when maxD = 0)

weighted aggregate (WORKLOAD_WEIGHTS_V1):
  WorkloadScore = round( 0.45·hoursIndex + 0.30·distanceIndex + 0.25·assignmentIndex )
                  clamped to 0–100
```

**Explainability (Phase H)** — each component's share of *this driver's* score:

```
contribHours       = 0.45·hoursIndex
contribDistance    = 0.30·distanceIndex
contribAssignment  = 0.25·assignmentIndex
raw                = contribHours + contribDistance + contribAssignment   (= score pre-round)

contribution% (per component) = round(contribX / raw × 100)   → e.g. "Jam 42% · Jarak 34% · Asg 24%"
```
Each driver card in the Driver Analytics view renders these three percentages as a segmented
bar + legend, and the raw indices in a tooltip. No black-box scoring.

**Per-driver utilization** (for the Utilization Ranking):
```
utilization = round( H / (officeHoursPerDay × daysWorked) × 100 )     (null if daysWorked = 0)
```
`officeHoursPerDay` derives from the existing `operations.workStartMins/workEndMins` (Jam Operasional)
— the SAME window the v1.16.4.7 overtime engine uses. No new setting.

## 3. Weights & Justification (Phase A/B)

`WORKLOAD_WEIGHTS_V1 = { hours: 0.45, distance: 0.30, assignments: 0.25 }` (sum = 1.0)

| Component | Weight | Justification |
|---|---|---|
| Actual Working Hours | **0.45** | Most direct, time-true measure of labour/workload; the metric HOS/driver-utilization analytics privilege. Highest weight. |
| Distance Travelled | **0.30** | Operational exposure/intensity (highway vs idle) that hours alone miss; primary exposure metric in logistics/safety. Second. |
| Completed Assignments | **0.25** | Dispatch/coordination load (fixed per-task overhead), but weakest effort proxy (ignores duration & distance). Lowest, non-trivial. |

The weights are a single named, exported constant (`workload-engine.js`) and are rendered into
both the dashboard note and the PDF note, so the formula is never a hidden magic number.

## 4. Ranking Engine (Phase C)

All ranks are derived from the Workload Score (NOT assignment count):

- **Driver Paling Aktif** = highest Workload Score.
- **Driver Beban Kerja Tertinggi** = highest Workload Score (== Paling Aktif, by definition).
- **Driver Beban Kerja Terendah** = lowest Workload Score among scored drivers (null if < 2 drivers).
- **Driver Utilization Ranking** = sorted by per-driver working-hour **utilization** (primary),
  Workload Score as tiebreak. Exact calc in §2.

Ordering is deterministic: score desc → hours desc → name asc.

## 9. Alias Investigation Findings (Phase J — root cause only, NO fix)

**Symptom:** Analytics Driver → Deteksi Duplikasi → Gabungkan Alias Tujuan → Nilai Kustom →
toast **"Gagal menyimpan alias."**

**Root cause:** the alias map is written to Firebase RTDB keyed by a **normalized name that is not a
legal RTDB key**. `_normDestKey()` lowercases/trims/collapses but does **NOT** strip the characters
RTDB forbids in keys — `.`, `#`, `$`, `/`, `[`, `]`. When the destination contains any of these
(e.g. `"Jl. Sudirman"`, `"No.5"`, `"PT. ABC"`, `"Gd A/B"`), the resulting object key is illegal and
the Firebase SDK throws synchronously inside `set()`, which is caught and surfaced as the generic toast.

**Failure trace:**
- **File:** `js/app.js`
  - `_normDestKey()` (re-imported from `js/analytics/analytics-engine.js`, lines 27–36) — produces the offending key; does not sanitize `. # $ / [ ]`.
  - `btnAliasSave` click handler (≈ line 6299–6328) — computes `aliasKey = _normDestKey(aliasName)` and calls `_saveAnalyticsAlias()`; its `catch` shows "Gagal menyimpan alias."
  - `_saveAnalyticsAlias(type, aliasKey, canonical)` (≈ line 6096) — builds `updated = { ...current, [aliasKey]: entry }` and calls `updateSetting('analyticsAliases.${type}', updated)`.
- **File:** `js/settings-store.js` → `updateSetting()` (line 130) → `storeFirebaseData()`.
- **File:** `js/firebase.js` → `storeFirebaseData()` (line 367) → `set(dbRef, value)` — **throws here** when any key in `value` contains an illegal RTDB character.

**Why "Nilai Kustom" appears in the repro but is NOT the trigger:** the failing element is the object
**key** (always the *detected name's* normalized form), not the canonical **value**. The same name
fails via the "A"/"B" radio choices too; the custom path is simply how the tester reproduced it.
(A custom canonical *value* may itself contain `.`/`/` harmlessly — values are unconstrained.)

**Fix deferred to v1.16.4.10** (per scope). Likely direction: sanitize/encode the alias key
(strip or percent-encode `. # $ / [ ]`) before writing, with a one-time migration of existing maps —
**not implemented here**.
