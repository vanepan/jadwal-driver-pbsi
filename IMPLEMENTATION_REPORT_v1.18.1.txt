1. ARCHITECTURE SUMMARY
════════════════════════════════════════════════════════════════════════════════

DESIGN PHILOSOPHY:
• Embedded maintenance model (no separate collection) — maintenance records 
  stored as arrays within vehicle documents
• Single Source of Truth: Vehicle Store (vehicles/{vehicleId}/maintenanceRecords/)
• PURE service layer (all functions Node-testable, no DOM/Firebase coupling)
• Unified Scoring integration (maintenance health component: 0.05 weight)

LAYER ARCHITECTURE:
├─ Configuration Layer (maintenance-config.js)
│  └─ 13 categories, 4 types, 4 statuses, 4 impacts + registry lookups
├─ Service Layer (maintenance-service.js)
│  └─ Validation, normalization, timeline, summary, health scoring
├─ Analytics Layer (maintenance-analytics.js)
│  └─ KPI aggregations, distributions, trends, rankings
├─ Data Store (vehicles-store.js CRUD)
│  └─ Add/Update/Delete/Get maintenance records with Firebase sync
├─ Asset Service Integration (vehicle-asset-service.js)
│  └─ Computes maintenance into vehicle health (now 4-component: operational/legal/documents/maintenance)
└─ UI Layer (fleet-dashboard, vehicle-detail-drawer, app.js)
   └─ Dashboard (6 KPI + 3 charts), Drawer (7 sections with maintenance), App wiring

MAINTENANCE CATEGORIES (13):
• periodic-service, oil-change, tire-replacement, brake, battery, engine, 
  transmission, suspension, electrical, body-repair, air-conditioning, 
  inspection, other

MAINTENANCE TYPES (4): preventive, corrective, emergency, predictive
MAINTENANCE STATUSES (4): planned, in-progress, completed, cancelled
MAINTENANCE IMPACTS (4): minor, medium, major, critical

HEALTH WEIGHTS (v1.18.1 UPDATE):
• operational: 0.40 (vehicle status)
• legal: 0.35 (STNK/tax/insurance validity)
• documents: 0.20 (down from 0.25)
• maintenance: 0.05 (NEW)

FIREBASE PATH STRUCTURE:
vehicles/{vehicleId}/maintenanceRecords/{recordId}
  └─ Fields: id, vehicleId, date, daysAgo, category, categoryLabel, type (auto-derived),
             typeLabel, impact, impactLabel, impactTone, status, statusLabel, 
             statusTone, workshopId, workshopName, officer, description, notes,
             odometer, odometerEstimated, cost, costDisplay, costBreakdown,
             createdAt, updatedAt, mediaIds (reserved), inspectionReport (reserved),
             reminderStatus (reserved), reminderSentDate (reserved),
             recommendedIntervalKm, recommendedIntervalDays

7. RISKS & MITIGATION
════════════════════════════════════════════════════════════════════════════════

RISK: Embedded maintenance model scalability
├─ Fleet size: <15 vehicles (constraint)
├─ Expected max records: ~150 per vehicle (acceptable)
├─ If scale exceeds: Archive migration planned for v1.19+
├─ Current impact: NONE (fleet size controlled)
└─ Mitigation: Monitor maintenance.totalRecords in analytics; log to Sentry if > 200

RISK: Workshop name inconsistencies (duplicate analytics)
├─ Problem: Free-text names can duplicate ("Bengkel PBSI" vs "bengkel pbsi")
├─ Current solution: Manual data quality (admin responsibility)
├─ Future solution: Workshop registry + validation (v1.18.2+)
├─ Current impact: MINOR (analytics may show duplicates)
└─ Mitigation: Admin guidance in UI; search includes all forms

RISK: Health score weight rebalancing
├─ Change: documents 0.25 → 0.20, +maintenance 0.05
├─ Impact: All existing health scores will shift slightly
├─ Direction: Vehicles with poor maintenance history will score lower
├─ Backward compatibility: Scores computed fresh each load (no stored history)
└─ Mitigation: Admin notified; scores recalculate automatically

RISK: Firebase quota consumption
├─ New writes: addMaintenanceRecord (1 write per record)
├─ New reads: computeMaintenanceHealth (included in vehicle normalization)
├─ Fleet size: <15 vehicles
├─ Max records: ~150 per vehicle → ~2,250 total reads on dashboard load
├─ Current quota: Typical Spark plan = 50K reads/day
└─ Mitigation: Well within limits; no action needed

RISK: Missing import in app.js maintenance handlers
├─ Current: validateMaintenanceRecord, normalizeMaintenanceRecord imported
├─ Missing: Actual maintenance form UI + event handlers (planned v1.18.2)
├─ Impact: Maintenance data structure in place, but no UI to add records yet
└─ Mitigation: Empty maintenance records OK (dashboard shows 0); UI added in next sprint

CONSTRAINT: Excluded from v1.18.1 per user requirement
├─ ❌ Vehicle Gallery (reserved for future)
├─ ❌ OCR (reserved for future)
├─ ❌ AI Predictive (reserved for future)
├─ ❌ Reminder Engine (reserved for future)
├─ ❌ Media Platform (reserved for future)
└─ ✅ All constraint boundaries respected
